# ロードマップ経由地入力機能 詳細仕様書

## 1. 目的

TaskSchedulerV3 のロードマップ登録・編集機能を、Google マップのルート入力UIのように、
**1画面・少タップ・その場編集・即追加・即削除・並び替え可能** な形式へ改善する。

本仕様書は、単なる画面見た目ではなく、**完璧に動作させるために必要な内部仕組み、状態管理、保存方式、描画方式、バリデーション、通知連携、一覧連携、詳細画面連携、エッジケース対策** まで定義する。

---

## 2. 完成イメージ

ユーザーはタスク詳細画面から「ロードマップ編集」を開き、以下のような1画面で操作できる。

- 先頭に START（元タスク）が表示される
- その下に中間ステップが縦並び表示される
- 最後尾が自動的に GOAL 表示になる
- 各行で直接タイトル編集できる
- 各行の日付をワンタップで変更できる
- `⊕ ステップを追加` でその場に新規行追加
- `✕` で即削除
- 長押しドラッグで並び替え
- 完了済みは視覚的に区別される
- 保存時に通知と一覧表示が正しく更新される

---

## 3. この機能を完璧に動作させるために必要な仕組み

この機能を安定して成立させるには、以下 8 層の仕組みが必要である。

1. **データ層の整合性維持**
2. **編集用UI状態の分離管理**
3. **インライン入力UI**
4. **並び替えとGOAL自動判定**
5. **保存時の差分反映**
6. **通知再登録**
7. **一覧・詳細画面との同期**
8. **異常操作・中断操作の保護**

以降、それぞれ詳細に定義する。

---

## 4. データ構造要件

### 4.1 前提エンティティ

既存の `RoadmapStep` は以下想定で運用する。[page:2]

| フィールド | 型 | 用途 |
|---|---|---|
| `id` | Long | ステップ識別子 |
| `taskId` | Long | 親タスクID |
| `title` | String | ステップ名 |
| `targetDate` | Long? | ステップ日付 |
| `isCompleted` | Boolean | 完了状態 |
| `order` | Int | 並び順 |

### 4.2 必須ルール

- 1つの `taskId` に対して複数の `RoadmapStep` を持てること。[page:2]
- 一覧表示順は必ず `order ASC` とすること。[page:2]
- GOAL専用フラグは持たず、**末尾要素をGOALとして扱う** こと。[page:2]
- START は `Task` 本体であり、`RoadmapStep` レコードを作らないこと。[page:1][page:2]
- `order` は 0,1,2... の連番に再採番できること。[page:2]

### 4.3 データ層で必要な仕組み

#### 必須DAO機能

- `getStepsForTask(taskId)`：対象タスクの全ステップ取得。[page:2]
- `insertStep(step)`：1件追加。[page:2]
- `updateStep(step)`：1件更新。[page:2]
- `deleteStep(step)`：1件削除。[page:2]
- `replaceSteps(taskId, steps)`：対象タスクのステップを差分または全置換保存する仕組みが必要。[page:2]
- `reindexOrders(taskId)`：削除・並び替え後に `order` を再採番する仕組みが必要。[page:2]

#### 望ましいDAO補助

- `deleteStepsByTaskId(taskId)`
- `insertSteps(steps: List<RoadmapStep>)`
- `updateSteps(steps: List<RoadmapStep>)`

完全動作のためには、**1件ずつの保存だけでなく、編集画面終了時に一括整合化できる保存ルート** が必要である。[page:2]

## 5. UI状態管理の仕組み

### 5.1 DBエンティティを直接編集してはいけない

画面上で直接 `RoadmapStep` を編集対象にすると、以下の問題が起きる。

- 未保存入力が即DBへ反映される
- 並び替え途中で不整合が起きる
- 取消や戻る操作が困難になる
- 新規追加行（まだDB未保存）を扱いにくい

そのため、**編集画面専用のUIモデル** を持つ必要がある。[page:0][page:1]

### 5.2 必須UIモデル

```kotlin
data class RoadmapStepEditorItem(
    val localId: String,
    val dbId: Long?,
    val taskId: Long,
    val title: String,
    val targetDate: LocalDate?,
    val isCompleted: Boolean,
    val order: Int,
    val isDeleted: Boolean = false,
    val isDirty: Boolean = false,
    val isNew: Boolean = false
)
```

### 5.3 各フィールドの役割

| フィールド | 理由 |
|---|---|
| `localId` | 未保存行もComposeで安定キー管理するため |
| `dbId` | 既存DB行と新規行を区別するため |
| `taskId` | 対象タスクを保持するため |
| `title` | 入力中の文字列保持 |
| `targetDate` | DatePickerとの連携用 |
| `isCompleted` | 完了済み見た目と保存連携 |
| `order` | 表示順・保存順管理 |
| `isDeleted` | 即削除UIでも保存前に差分管理するため |
| `isDirty` | 変更有無の判定に必要 |
| `isNew` | insert対象判定に必要 |

### 5.4 ViewModelで必要な状態

`RoadmapEditViewModel` には最低限以下が必要である。[page:1]

- `task: StateFlow<Task?>`
- `editorItems: StateFlow<List<RoadmapStepEditorItem>>`
- `hasUnsavedChanges: StateFlow<Boolean>`
- `focusedItemLocalId: StateFlow<String?>`
- `pendingDeleteUndoItem: StateFlow<RoadmapStepEditorItem?>`
- `isSaving: StateFlow<Boolean>`
- `errorMessage: StateFlow<String?>`

これがないと、**画面復帰、Undo、保存ボタン活性制御、編集中フォーカス維持** が不安定になる。[page:1]

## 6. 入力UIの仕組み

### 6.1 インライン編集方式

各ステップ行は別ダイアログではなく、その場で編集できる必要がある。[page:0]

#### 必須部品

- `BasicTextField` または `TextField`
- 日付チップボタン
- 削除ボタン
- ドラッグハンドル
- ステップ種別アイコン

### 6.2 行構成

```
[⠿] [○/◎] [タイトル入力欄................] [4/20] [✕]
```

### 6.3 入力時の仕様

- 行タップで入力欄にフォーカス。[page:0]
- 新規追加直後は自動フォーカス。[page:0]
- キーボードの `完了` でフォーカスを外す。[page:0]
- フォーカスアウトでViewModelに即反映。[page:1]
- 空文字は保存時に弾く、または削除扱いにする必要がある。[page:1]

### 6.4 文字入力で必要な保護

- 連続入力でもラグが出ないこと
- 1文字ごとのDB保存は禁止、UI状態だけ更新すること
- 200文字上限など必要なら既存Taskタイトル基準に合わせること
- 改行は不可、1行固定が望ましい

---

## 7. 日付入力の仕組み

### 7.1 日付変更UI

各行の `4/20` のような日付バッジをタップするとDatePickerが開く必要がある。[page:0]

### 7.2 必須要件

- 日付未設定も許容すること
- 日付設定済みなら `M/d` で表示すること
- 日付未設定なら `日付` と表示してもよい
- START日より前の日付も許容すること
- ステップ同士の日付逆転も許容すること

この機能は進行管理ではなく、**柔軟な見通し管理** が主目的のため、日付順強制はしない。[page:1][page:2]

### 7.3 保存時ルール

- DatePicker確定時にUI状態更新
- DB保存は画面全体保存時にまとめて行う
- 日付削除機能が必要なら「日付クリア」を用意する

---

## 8. ステップ追加の仕組み

### 8.1 追加動作

`⊕ ステップを追加` を押すと、末尾に未保存の新規行を1件追加する。[page:0]

### 8.2 自動処理

- `localId` をユニーク生成
- `dbId = null`
- `isNew = true`
- `isDirty = true`
- `order = 現在末尾+1`
- `targetDate = task.date` を初期値にしてよい
- 追加後にその行へ自動スクロール
- 追加後にその行へ自動フォーカス

### 8.3 連打対策

- 追加ボタン連打で空行が大量生成されないよう制御する
- 末尾の空タイトル新規行が存在する間は追加ボタンを無効化、またはその行へフォーカス移動する

この仕組みがないと、簡単操作にした結果、**空ステップだらけになる**。[page:0]

---

## 9. 削除の仕組み

### 9.1 削除方式

右端の `✕` タップで即削除する。[page:0]

### 9.2 内部処理

#### 新規未保存行の場合
- リストから即除去する
- DB操作不要

#### 既存保存済み行の場合
- UI上は即非表示
- 内部では `isDeleted = true` にして保持してもよい
- 保存時に `DELETE` 実行

### 9.3 Undo推奨

完全動作を目指すなら、削除後にSnackbarで `元に戻す` を出せる仕組みが望ましい。

- `pendingDeleteUndoItem` を保持
- SnackbarのActionで復元
- 一定時間経過または保存時に確定削除

これにより、確認ダイアログなしでも操作性と安全性を両立できる。[page:1]

---

## 10. 並び替えの仕組み

### 10.1 目的

Googleマップ同様、順序変更をドラッグ操作だけで終わらせる必要がある。[page:0]

### 10.2 必須実装要件

- ComposeのLazyColumn上でドラッグ並び替えできること
- ドラッグ中に対象行の見た目が変わること
- ドロップ時に `order` を再計算すること
- 保存前のUI状態にも即反映すること

### 10.3 GOAL自動判定

GOAL専用データは持たず、**並び替え後の末尾行をGOAL表示** とする。[page:2]

つまり必要なのは以下。

- `isGoal = index == visibleItems.lastIndex`
- 並び替え・削除・追加のたびに再判定

### 10.4 並び替えで必要な注意

- 完了済み行も並び替え可能かを明確化する必要がある
- 推奨は「完了済みも移動可」だが、保存時に整合を取る
- ドラッグ対象のキーは `localId` を使うこと

`dbId` だけをキーにすると新規行が壊れる。[page:1]

---

## 11. START / 中間 / GOAL 表示ロジック

### 11.1 START

- `Task` 本体のタイトルと日付を表示する
- 編集不可
- アイコンは `▶`
- ロードマップの開始点として常に先頭固定

### 11.2 中間

- 末尾以外のステップは `○`
- 編集可
- 削除可
- 並び替え可

### 11.3 GOAL

- 最後尾ステップのみ `◎`
- 編集可
- 削除可
- 並び替えにより別行へ移動しうる

### 11.4 ステップ0件時

- STARTのみ表示
- その下に `⊕ ステップを追加`
- GOAL表示はまだ存在しない

この0件状態の扱いを明確にしないと初回登録時にUIが崩れる。[page:0]

## 12. 保存の仕組み

### 12.1 保存は一括で行う

完全動作のためには、行ごと自動保存ではなく、**編集画面全体を一括保存** する方式が必要である。[page:1][page:2]

理由：

- 並び替えと削除と編集が同時に起きる
- 個別即保存だと順序不整合が起きやすい
- 通知再登録タイミングをまとめやすい

### 12.2 保存前バリデーション

保存前に以下を検査する必要がある。

- 空タイトルの既存行が残っていないか
- 重複orderがないか
- taskId不一致がないか
- 削除対象と更新対象が重複していないか

### 12.3 保存処理順

推奨順は以下。

1. 現在表示中の有効行を抽出
2. orderを0から再採番
3. 削除対象をDELETE
4. 新規行をINSERT
5. 既存行をUPDATE
6. 再取得してViewModel更新
7. 通知再スケジュール
8. 詳細画面へ戻る

### 12.4 トランザクション

保存処理は**必ずDBトランザクション**で行う必要がある。[page:2]

そうしないと、例えば「3件更新中に1件だけ失敗」で順序や通知が壊れる。

## 13. 通知連携の仕組み

ロードマップ機能は、見た目だけでなく通知と連携して初めて完成する。[page:1]

### 13.1 必須要件

- ステップ追加時、対象step通知が登録される
- ステップ日付変更時、既存通知をキャンセルして再登録する
- ステップ削除時、対象通知をキャンセルする
- 並び替えだけなら通知時刻が変わらない場合は再登録不要でもよい
- 保存時に対象task配下の通知を整合化する

### 13.2 必要な内部キー

通知を安定動作させるには、**step単位の一意な通知ID** が必要である。[page:3]

例：

```kotlin
notificationId = (taskId * 1000 + stepOrder).toInt()
```

ただし `order` は変わるため、理想は `step.id` ベースで通知IDを作ること。

### 13.3 推奨方式

- DB保存後に `step.id` が確定する
- `step.id` を元に PendingIntent requestCode を生成する
- 保存完了後、全ステップ通知を再構築する

この方式なら並び替えや再採番でも通知が壊れにくい。[page:3]

## 14. 一覧画面との連携

TaskSchedulerV3 では、ロードマップは一覧表示の次実行段階に反映される設計である。[page:0][page:1]

### 14.1 必須連携

保存後に一覧側で以下が正しく変わる必要がある。

- 次のアクティブステップ名
- 次の表示日付
- `🛣️` の表示
- `【締切】` など接頭辞つきタイトル
- 進捗率

### 14.2 必要な仕組み

- 詳細画面へ戻った後に一覧用Flowが最新DBを再購読していること
- `activeRoadmapStepId` または現在未完了先頭stepを再計算すること
- ステップ0件なら元Task表示に戻れること

ロードマップ編集だけ完成しても、一覧が更新されなければユーザーは壊れていると感じる。[page:1]

## 15. 詳細画面との連携

### 15.1 必須要件

ロードマップ編集後、`TaskDetailScreen` 側で以下が更新される必要がある。[page:2]

- 縦タイムラインの順序
- 各ステップ名
- 各ステップ日付
- GOAL位置
- 完了済み見た目

### 15.2 復帰方法

- 保存成功後に `onBack()` で戻るだけでなく、詳細画面が再読み込みされる仕組みが必要
- `collectAsState()` でDB Flowを監視していれば自然更新できる

---

## 16. 完了済みステップの扱い

### 16.1 見た目

完了済みは以下の見た目が必要である。

- アイコン色変更
- 縦線色変更
- タイトル色変更
- 取り消し線は任意

### 16.2 編集可否

仕様で明確化が必要だが、実用上は以下がよい。

- 完了済みでもタイトル編集可
- 完了済みでも日付変更可
- 完了済みでも並び替え可
- 完了済みでも削除可

ただし、保存時に通知は再登録しないなどの制御が必要である。[page:3]

---

## 17. バックキー・画面離脱の保護

### 17.1 必須保護

インライン編集は便利だが、誤って戻ると入力が消えやすい。
そのため、未保存変更がある場合の離脱保護が必要である。[page:1]

### 17.2 推奨仕様

`hasUnsavedChanges == true` の時：

- バックキー押下
- 上部戻るボタン押下

この両方で確認ダイアログを出す。

文言例：

- 「変更を保存せずに閉じますか？」
- ボタン：`キャンセル` / `破棄して閉じる`

---

## 18. 画面回転・プロセス再生成対策

完璧動作を目指すなら、端末回転やメモリ再生成でも編集内容が飛ばない仕組みが必要である。

### 必須仕組み

- `SavedStateHandle` に編集対象taskIdを保持
- 可能なら `editorItems` を復元可能にする
- 最低限、未保存変更中の回転を避けるか、再読込手順を定義する

Composeだけに任せると、場合によって入力中状態が飛ぶことがある。[page:1]

---

## 19. エッジケース仕様

### 19.1 空タイトルで保存

- 新規空行は保存しない
- 既存行を空にした場合はエラー表示または削除扱い

### 19.2 ステップ0件

- STARTのみ表示
- 一覧は元Taskをそのまま表示

### 19.3 ステップ1件

- その1件がGOALになる

### 19.4 すべて削除

- STARTのみへ戻る
- activeRoadmapStepId を null に戻す必要がある
- step通知は全キャンセル

### 19.5 同日ステップ複数

- 許容する
- 表示順は `order` 優先

### 19.6 過去日付

- 許容する
- 通知は即時または登録しないなど既存通知仕様に合わせる

### 19.7 並び替え中に削除

- ドラッグ中操作を無効化するか、ドラッグ終了後のみ削除可にする

完全動作・保守性確保のため、`RoadmapEditScreen.kt` を巨大化させず、最低限以下へ分割する必要がある。[page:0][page:1]

- `RoadmapEditScreen`

- `RoadmapTimelineIcon`

この分割がないと、入力・フォーカス・ドラッグ・日付変更・削除の相互干渉を管理しにくい。

## 22. 非機能要件

### 22.1 操作感

- 新規追加後 200ms 以内に入力可能状態になること
- 入力中のスクロール・フォーカス遷移が破綻しないこと
- 並び替えがカクつかないこと

### 22.2 視認性

- 1行あたりの高さは詰めすぎない
- 日付バッジと削除ボタンが誤タップしにくいこと
- GOALが一目で分かること

### 22.3 互換性

- 既存ロードマップデータを壊さないこと
- 既存通知機能と干渉しないこと
- 無期限予定・繰返し予定にも必要なら適用可否を明確にすること

## 24. 完全動作の判定基準

この機能が「完璧に動作している」と判定する条件は以下である。

1. ステップ追加が1タップで始められる
2. テキスト編集がその場でできる
3. 日付変更がすぐできる
4. 削除が1タップでできる
5. 並び替えで順序が正しく保存される
6. GOALが常に末尾に自動追従する
7. 保存でDBが壊れない
8. 保存後に詳細画面へ即反映される
9. 保存後に一覧表示へ即反映される
10. 通知が追加・更新・削除に追従する
11. バック操作で未保存変更を保護できる
12. 0件・1件・複数件・削除しきり・日付逆転でも破綻しない

## 25. 結論

この機能は、単に `RoadmapEditScreen` の見た目を Google マップ風に変えるだけでは不十分である。
完全に動かすには、**編集専用UI状態、差分保存、一括トランザクション保存、GOAL自動判定、通知再構築、一覧・詳細同期、未保存保護、エッジケース対策** をセットで設計する必要がある。[page:0][page:1][page:2][page:3]

見た目だけ先に作ると高確率で「入力は楽になったが保存で壊れる」「並び替え後に一覧が狂う」「通知がズレる」という不具合が出るため、本仕様書の仕組みを一式組み込む前提で実装すること。[page:0][page:1][page:2][page:3]
