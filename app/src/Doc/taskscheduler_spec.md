# TaskSchedulerV3 追加機能仕様書

本仕様書は、既存の TaskSchedulerV3 ローカルコードに対して、以下の 2 機能を追加するための詳細仕様を定義する。

- 機能A: 親子関係ベースの関連予定機能 [完了]
- 機能B: ロードマップ機能 [完了]

本仕様書は、ユーザーが指定した 13 項目のみを対象とし、それ以外の仕様変更は行わない。
既存 GitHub リポジトリのローカルコードは GitHub と一致している前提とする。[page:5][page:6][page:7][page:8][page:9][page:10]

---

## 1. 目的

既存アプリは予定一覧・詳細・完了管理を中心とした構成であり、タスク同士の前後関係や段階的な進行を表現する仕組みが弱い。[page:5][page:9][page:10]
本追加仕様では、単発予定の管理だけでなく「ある予定から派生する次の予定」および「1つの予定の中にある段階的進行」を扱えるようにする。

---

## 2. 今回追加する機能の全体像

### 2.1 機能A: 親子関係ベースの関連予定機能

既存の予定同士を単純な双方向リンクではなく、親予定と子予定の関係で管理する。

例:
- 親予定: 4/1 いちご狩り募集締め切り
- 子予定: 4/10 いちご狩り参加者案内配布

この場合、4/10 の予定は 4/1 の予定に関連するのではなく、4/1 を親、4/10 を子として登録する。

### 2.2 機能B: ロードマップ機能

1つの予定に対して、開始からゴールまでの中間目標を縦型タイムラインで保持する機能を追加する。
ロードマップを使うことで、新規予定を大量に作らなくても、1つの予定の進行段階を表現できるようにする。

---

## 3. 対象範囲

本仕様書で対象とする追加・変更は次の通り。

1. 親子関係による関連予定管理
2. 関連予定の後付け登録
3. 詳細画面での親子予定一覧表示
4. 一覧画面での関連件数表示
5. ロードマップ編集機能
6. ロードマップの縦型タイムライン表示
7. ロードマップ完了による進捗率反映
8. ロードマップ完了による通知連携
9. ロードマップのドラッグ＆ドロップ並び替え
10. ロードマップ進行に応じた予定一覧の表示切り替え
11. 一覧カードの左右スワイプアクション
12. 既存優先度フィルター（高・中・低）の削除
13. 予定種別ごとの絵文字付与

以下は今回 **対象外** とする。

- 関連予定候補の自動サジェスト
- 関連あり／ロードマップありフィルターの追加
- 関連予定の一覧画面での色線やグループ色表現
- 残○日バッジの色変更
- 今回指定されていない既存UI改善
- 今回指定されていない既存データ構造の大幅整理

## 4. 現状前提

既存コード上、Task、TaskRelation、AppDatabase、TaskDao、ScheduleListViewModel、TaskRowItem などの構造が存在し、一覧画面とデータベース連携の基盤はすでにある。[page:8][page:9][page:10][page:11][page:12]
したがって、本追加では既存構造を活かしながら、最小限の拡張で機能を組み込む。

## 5. 機能A: 親子関係ベースの関連予定機能

## 5.1 要件概要

- 予定同士を双方向関連ではなく親子関係で保持する
- 親予定は複数の子予定を持てる
- 子予定は原則として親予定を 1 件持つ
- 後から親子関係を設定・変更可能
- 詳細画面で親予定・子予定を表示可能にする
- 一覧画面に関連件数を小さく表示する
- 左スワイプで関連付けを開始できる

## 5.2 データモデル仕様

既存の `TaskRelation` がある場合でも、双方向関連として使っているなら用途を変更する。[page:8]
今回の仕様では、関係性は親子で定義する。

### 5.2.1 推奨構造

#### 案1: Task 側に parentTaskId を追加

- `Task.parentTaskId: Long?`
- 親を持たない予定は null
- 子予定は親 Task の id を保持

この方式は単純で、1子予定につき親1件の制約を自然に表現できる。
今回の要件に最も適している。

#### 案2: TaskRelation を親子定義用テーブルとして再利用

- `parentTaskId`
- `childTaskId`
- relationType = PARENT_CHILD

ただし今回の要件では 1 子予定 1 親予定で十分なため、Task へ `parentTaskId` を追加する案を優先する。

### 5.2.3 採用仕様

最終的に **Task に `parentTaskId` を追加する** 方式を採用した。

### 5.2.4 Task 拡張項目 (実装値)

- `parentTaskId: Int? = null`
- `roadmapEnabled: Boolean = false`
- `activeRoadmapStepId: Int? = null`

※ `taskKind` は絵文字表示やロードマップ／通常予定区別に利用する。

## 5.3 親子関係登録仕様

### 5.3.1 新規予定作成時

- ユーザーは予定作成画面で「親予定を選択」できる
- 選択した場合、新規予定の `parentTaskId` にその親予定 id を保存する
- 未選択なら通常予定として保存する

### 5.3.2 後から関連付け

- 既存予定詳細画面、または一覧画面の左スワイプ経由で「親予定に設定」または「子予定を追加」を行える
- 既存の子予定に対して親変更も可能
- 自分自身を親にはできない
- 循環参照は禁止する
  - A の親を B にした状態で、B の親を A にすることは不可
  - A → B → C のとき、C を A の親にすることも不可

## 5.4 一覧画面表示仕様

各予定カードに小さく関連件数を表示する。

表示ルール:
- 子を 1 件以上持つ親予定: `関連1`, `関連2` のように表示
- 親を持つ子予定: `親あり` または `関連1` のどちらかに統一
- 本仕様では見た目を増やしすぎないため、**親・子ともに `関連n` 表示に統一**してよい
- 関連がなければ非表示

## 5.5 詳細画面表示仕様

予定詳細画面に「関連予定」セクションを追加する。

表示内容:
- 親を持つ場合: 親予定を 1 件表示
- 子を持つ場合: 子予定一覧を表示
- タップすると対象予定の詳細画面へ遷移

表示順:
- 親予定: セクション最上部
- 子予定: 日付昇順

## 5.6 スワイプ仕様

### 5.6.1 左スワイプ

- アクション名: 関連付け
- 動作: 対象予定を起点として親選択または子追加ダイアログを開く
- 一覧画面から素早く親子設定できるようにする

### 5.6.2 右スワイプ

- アクション名: 完了
- 既存の完了処理を呼び出す
- ロードマップ中の予定であれば、単純完了ではなく「現在アクティブなロードマップ段階を完了」する挙動に接続する

## 6. 機能B: ロードマップ機能

## 6.1 要件概要

- 予定詳細からロードマップ編集画面を開ける
- ロードマップは縦型タイムラインで表示する
- 開始日は予定自体の日付とする
- 最後に登録されたステップがゴールとなる
- ステップは追加・編集・削除・完了できる
- ステップはドラッグ＆ドロップで並び替えできる
- ステップ完了に応じて進捗率に反映する
- ステップ完了に応じて次に表示される一覧予定を差し替える
- ステップ期限で通知を発生させる

## 6.2 ロードマップ構造

### 6.2.1 新規テーブル `RoadmapStep`

推奨フィールド:

- `id: Long`
- `taskId: Long`
- `title: String`
- `date: LocalDate` または既存形式に合わせた日付型
- `stepType: RoadmapStepType`
- `sortOrder: Int`
- `isCompleted: Boolean`
- `completedAt: Long?`
- `notificationEnabled: Boolean`
- `notificationRequestCode: Int?`

### 6.2.2 stepType

- `START`
- `MILESTONE`
- `GOAL`

ただし UI 上は START/GOAL を完全自動でもよい。

自動判定ルール:
- 一番先頭: START
- 一番最後: GOAL
- 間: MILESTONE

## 6.3 開始日・ゴール日定義

ユーザー指定に従い、ロードマップが 1 件でも存在する場合:

- Task に元々登録されている予定日を開始日とみなす
- ロードマップで最後に並んでいるステップの日付をゴールとみなす
- ロードマップ途中のステップは中間目標とみなす

つまり、元の Task 自体が起点であり、ロードマップはその Task を拡張する段階情報である。

## 6.4 ロードマップ表示仕様

詳細画面の「ロードマップ」セクションで縦タイムライン表示する。

表示イメージ:

```text
◉ 4/5  高知旅行募集案内配布（START）
│
○ 4/12 参加者確認（未完了）
│
○ 4/20 締切（未完了）
│
◎ 5/3  当日（GOAL）
```

状態ごとの表示:
- 完了済み: 塗りつぶし・チェック付き
- 進行中: 強調表示
- 未完了: 通常表示
- GOAL: 終点アイコンで強調

## 6.5 ロードマップ編集仕様

編集画面で可能な操作:

- ステップ追加
- ステップ編集
- ステップ削除
- ステップ完了切替
- ドラッグ＆ドロップ並び替え

### 6.5.1 並び替え仕様

- ドラッグ＆ドロップ後、`sortOrder` を再採番する
- 並び順が変わった結果、最後尾ステップが自動的に GOAL になる
- 並び替え後に日付の逆転が起きても保存は可能とするか要検討だが、本仕様では **保存可能** とする
- ただし、一覧表示や通知基準で混乱しないよう、実装側では保存時に「日付順にしなくてよい」ことを明示する

## 6.6 一覧画面への反映仕様

### 6.6.1 根本ルール

ロードマップを持つ Task は、一覧画面において常に「次に実行すべき段階」だけを表示する。

例:
- Task 本体: 4/5 高知旅行募集案内配布
- Step1: 4/12 参加者確認
- Step2: 4/20 締切
- Step3: 5/3 当日

表示遷移:

1. 初期表示
   - 一覧表示: `4/5 高知旅行募集案内配布`
2. 4/5 を完了
   - 一覧表示: `4/12 【参加者確認】高知旅行募集案内配布`
3. 4/12 段階を完了
   - 一覧表示: `4/20 【締切】高知旅行募集案内配布`
4. 4/20 段階を完了
   - 一覧表示: `5/3 【当日】高知旅行募集案内配布`
5. 最終段階完了
   - Task 全体を完了扱いにする

### 6.6.2 一覧表示用タイトル生成

ロードマップ進行中の一覧タイトルは以下で生成する。

- 開始段階: `元のタスクタイトル`
- 中間・最終段階: `【ステップ名】元のタスクタイトル`

例:
- `【参加者確認】高知旅行募集案内配布`
- `【締切】高知旅行募集案内配布`

### 6.6.3 色付け仕様

`【参加者確認】` や `【締切】` の接頭辞部分は色付けする。

ルール:
- START: 既存アクセント色
- MILESTONE: 青系または中性色アクセント
- GOAL: 緑系またはゴールド系アクセント

※ ただし、既存デザインとの整合を優先し、色の種類は増やしすぎない。

## 6.7 完了率への反映仕様

ロードマップを持つ Task の進捗率は、ロードマップ全体の完了率と連動する。

計算式:

```text
完了率 = 完了済み段階数 / 全段階数 × 100
```

ここでの全段階数には、開始段階としての Task 本体も含める。

例:
- 開始 + 3 ステップ = 全4段階
- 1段階完了 → 25%
- 2段階完了 → 50%
- 3段階完了 → 75%
- 4段階完了 → 100%

既存の進捗表示バーと数値へこの計算結果を反映する。[page:10]

## 6.8 完了操作仕様

### 6.8.1 Task 本体が未完了のとき

- 通常の完了操作は「開始段階の完了」とみなす
- その後、activeRoadmapStepId を次の未完了ステップへ進める
- 一覧表示内容を次段階へ差し替える

### 6.8.2 中間段階がアクティブなとき

- 完了操作で当該ステップを完了
- 次ステップを active にする
- 一覧表示を更新する

### 6.8.3 GOAL 完了時

- Task 全体を complete = true にする
- 一覧から完了済み扱いとして処理する

## 6.9 通知仕様

ロードマップの各ステップは通知対象とする。

### 6.9.1 通知発火タイミング

- 各ステップの date を基準に通知設定可能
- 通常予定通知と同じ仕組みを利用する
- ただし通知文言はステップ名を含める

例:
- `【参加者確認】高知旅行募集案内配布 の予定日です`
- `【締切】高知旅行募集案内配布 の予定日です`

### 6.9.2 次段階切替後の通知

開始段階を完了した時点で、次にアクティブとなるステップの通知が有効であれば再スケジュール確認を行う。

### 6.9.3 削除時の通知

- ステップ削除時、そのステップ通知は必ずキャンセル
- 並び替えで日付変更した場合も再登録

## 9. 予定種別ごとの絵文字仕様

既存予定の見分けやすさ向上のため、各予定に絵文字を付与する。

### 9.1 提案する絵文字

- 通常予定: `📅`
- 無期限予定: `📝`
- 繰返し予定: `🔁`
- ロードマップ予定: `🛣️`
- 親予定: `🌱`
- 子予定: `↪️`

### 9.2 採用方針

一覧での主表示は「予定種別」に限定し、親子は補助表示に留める。
つまり主たる表示ルールは次の通り:

- 通常予定: `📅`
- 無期限予定: `📝`
- 繰返し予定: `🔁`
- ロードマップ予定: `🛣️`

補助表示:
- 親子関係がある場合のみ詳細や補助ラベルで `🌱`, `↪️` を利用してよい

### 9.3 優先順位

1. ロードマップ予定
2. 繰返し予定
3. 無期限予定
4. 通常予定

たとえば「無期限かつロードマップ」は基本的に仕様上避けるべきため、UI では最上位 1 種だけ表示する。

## 11. ViewModel / Repository 変更仕様

## 11.1 一覧用表示モデルの拡張

一覧に表示する 1 行のデータを、DB の Task そのままではなく、表示用 ViewData として組み立てる。

保持情報例:
- 実 Task id
- 表示タイトル
- 表示日付
- 表示進捗率
- 表示絵文字
- 関連件数
- active stage 情報
- スワイプアクション可否

## 11.2 完了処理分岐

`completeTask(taskId)` 的な既存処理がある場合、以下に分岐する。

- ロードマップなし → 既存完了処理
- ロードマップあり & 開始段階未完了 → 開始段階完了
- ロードマップあり & 中間段階進行中 → current step 完了
- ロードマップあり & GOAL進行中 → Task 全体完了

## 11.3 一覧更新

ロードマップ完了時は、単純に `isCompleted=true` とするのではなく、次の表示段階へ切り替わるよう状態管理する。

## 14. 受け入れ条件

以下を満たしたとき、本仕様の実装完了と判断する。

### 14.1 親子関係

- 新規予定作成時に親設定できる
- 後から親設定できる
- 詳細画面で親・子が表示される
- 一覧に関連件数が表示される
- 左スワイプで関連付け操作に入れる

### 14.2 ロードマップ

- 詳細画面からロードマップ編集画面を開ける
- ステップ追加・編集・削除・完了・並び替えができる
- 縦タイムラインで表示される
- Task 完了で次段階が一覧に表示される
- 接頭辞部分に色が付く
- 進捗率がロードマップ完了数に応じて変化する
